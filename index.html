# AutoFlag Install Steps Redesign — Spec Doc **Doc version:** v7 **Status:** Spec locked. Rev A + Rev B + 1.91 + 1.92 + Rev C (1.93) shipped. Rev D (mobile polish) and Rev E (reference photos when Phil delivers them) remain. **Last updated:** Session of 2026-05-07 (AF2). Base rev: Rev 1.93. Currently deployed: Rev 1.89. --- ## What this is Phil wants a step-by-step installer wizard inside AutoFlag that walks installers through the install procedure with forced sequencing, required photos, and structured measurement capture. Today the install-steps section in the Edit Pole modal is a flat checklist that doesn't enforce sequencing or capture structured data. This redesign makes it a real guided-install tool. Reference document: Phil's `AutoFlag__Installation_Steps.pdf` (4 pages: terminology reference, pre-install measurements, halyard setup pre-onsite, onsite install steps). Phil also pasted a longer text version that includes 22 numbered onsite install steps with detail (tape, splice, double-barrel rope clamp, actuators, magnets, app commissioning, etc.). --- ## Locked-in decisions ### Scope & sequencing **Templates are global per install-type.** One template for External, one for Internal, one for Cylinder. No per-customer variants. Templates live in Firebase under `state.procedures` and are edited via the existing admin UI (`openProceduresModal()`). **Source of truth for procedures = Firebase, edited via admin UI.** No code deploys needed for procedure tweaks. `seed_procedures.json` is the seed for new installs only. **Backwards compat = extend, don't refactor.** Existing pole `p.checklist` schema (`{ stepId: { checked, checkedAt, checkedBy, photo? } }`) stays — we add new optional fields like `inputValue` for steps that capture measurements, and add new sections to the procedure schema. Existing checklists keep working. ### Forced sequencing **Forced order, hard block.** Step N+1's checkbox is grayed out / disabled until step N is checked. Installer cannot skip ahead. Phil: "if installers start complaining we can change it." ### Photo enforcement **Required photos are blocked-until-uploaded, not flagged-and-warned.** On steps where photo is required, the checkbox is disabled until at least one photo is uploaded for that step. Photo upload UI sits next to the checkbox — single-tap from the field. One photo required minimum. More photos optional (matches Eats per-step photo pattern). Phil's reasoning: ties evidence to the step itself rather than relying on a final-completion gate. Worth the small workflow friction if it prevents "completed" steps with no evidence. Open to revisiting after field experience. ### Measurement capture **Two named measurements: A (halyard travel) and B (flag height).** Hard-coded — not a generic "input fields per step" capability. Captured on specific named pre-install steps. Stored on the pole record (probably `p.measurements = { A: "20 ft", B: "36 in" }` — exact field name TBD during data model work). **Display behavior:** A and B are captured during Pre-Install Measurements (Section 2). They are then **displayed (not re-entered)** in Halyard Setup (Section 3) instructions. Step text uses `{A}` and `{B}` placeholders that get substituted with the captured values at render time. Example: Halyard Setup Step 1 instruction text would read: *"Cut new halyard to 2 × A = **40 ft** (A was 20 ft)"* with the bolded value rendered live from the pole's captured A. ### Sectioned procedure structure Each procedure template has 4 sections in this order: 1. **Terminology Reference** — collapsible, no checkboxes, viewable for reference. Diagram + term labels (Truck, Rotating Truck Assembly, Flag Height B, Truck-to-Cleat A, Swivel Flag Clips, Halyard, Cleat, Flash Collar). For Internal Units there's also a "measure circumference" reminder. Default state: collapsed. 2. **Pre-Install Measurements** — onsite first visit. Captures A (halyard travel) and B (flag height). Includes the steps for attaching tape-measure reel, lowering flag, recording measurements, raising halyard, and texting Phil with location/measurements/photos. ~6 steps total. 3. **Halyard Setup Prior to Onsite** — back at the office, prep before second visit. Uses A and B values. Includes: cut halyard to 2×A, fold in half, attach top swivel clip 4" from bend on LEFT side, attach bottom swivel clip at B+3" below top clip, install rubber covers, install actuator/limit switch with 14 magnets, etc. ~9 steps total. 4. **Onsite Installation** — final visit. The 22 numbered steps from Phil's pasted spec: pre-install verification photo, mount AutoFlag device at 8'6", lower existing flag, halyard replacement (with the critical "don't let halyard slip off top pulley" warnings), splice, loop closure with double-barrel clamp, backup C-clamp, flag re-hang, 2nd actuator install, solar panel attach, app commissioning at app.api-autoflag.com/setup-device, test up/down, final photo. ~22 steps total. **Sections are visual groupings within the Edit Pole modal**, NOT separate pages. Phil revised this from his initial "separate pages" thought during spec discussion — see "Separate top-level Installs view" decision below for what IS a separate page. **Forced sequencing applies across sections** (you can't do "Onsite step 1" until "Halyard Setup step 9" is checked, etc.). Soft-sequencing alternative was considered and rejected — Phil chose hard sequencing. ### Top-level Installs view **New top-level button "Installs" on the row alongside Browse / Kanban / List / Map / Expand toggle.** Tapping it shows a read-only **template viewer** (NOT a workspace). Flow: 1. Tap **Installs** in the top tab bar 2. See 3 cards: External / Internal / Cylinder 3. Tap one → see that procedure's 4 sections as expandable groups 4. View steps, instructions, reference photos — read-only **Per-pole work stays in the Edit Pole modal** where it already is. The Installs top-level view is for training, reference, and admin authoring (admins can edit from there). Phil confirmed Option A (template-only viewer, not workspace). ### Terminology Reference UX Collapsible block at top of Pre-Install Measurements section in Edit Pole modal. Default state: **collapsed** (don't take up space for installers who already know the terms). One-tap expand to view the diagram and term labels. In the top-level Installs view, terminology section behaves the same — collapsible, default collapsed. ### Per-step required-input fields Hard-coded for now. Only the two pre-install measurement steps have input fields (A and B). No generic input-field-builder in admin UI for this release. Can add later if needed. --- ### Rev A data model (locked) **Procedure template shape** (one per install type at `autoflag/procedures/{external|internal|cylinder}`): ``` { sections: [ { id: "terminology", // stable section id title: "Terminology Reference", kind: "reference", // reference | steps collapsedByDefault: true, content: { // only for kind:"reference" diagramUrl: "", // optional reference photo terms: [ // optional labeled terms { label: "Truck", desc: "..." }, ... ] } }, { id: "preInstall", title: "Pre-Install Measurements", kind: "steps", collapsedByDefault: false, steps: [ { ...stepObject } ] }, { id: "halyardSetup", title: "Halyard Setup Prior to Onsite", kind: "steps", collapsedByDefault: false, steps: [ { ...stepObject } ] }, { id: "onsiteInstall", title: "Onsite Installation", kind: "steps", collapsedByDefault: false, steps: [ { ...stepObject } ] } ], updatedAt: , updatedBy: } ``` **Step object shape** (extends Rev 1.88 schema; existing fields preserved): ``` { id: "step__", // existing — unchanged title: "...", // existing instructions: "...", // existing — supports {A} and {B} placeholders photoUrl: "", // existing — single reference photo (legacy) requirePhoto: false, // existing — installer photo required to check off // NEW in Rev A: referencePhotos: [], // array of URLs, multiple reference photos for the step inputField: null // null OR { name: "A" | "B", label: "...", unit: "ft" | "in", placeholder: "" } } ``` **Section omission rule.** Cylinder template omits `terminology`, `preInstall`, and `halyardSetup` sections entirely — `sections` array contains only the `onsiteInstall` section. Renderer iterates `sections` and renders whatever is present; missing sections are simply absent. **Pole `checklist` extension** (existing field on each pole, currently `{ stepId: { checked, checkedAt, checkedBy, photoUrl } }`): ``` checklist: { : { checked: bool, // existing checkedAt: , // existing checkedBy: , // existing checkedByName: "...", // existing photoUrl: "...", // existing — installer's photo for this step inputValue: "20 ft" // NEW — only present on steps with inputField, captures A or B } } ``` A and B values are read at render time from the pole's checklist by finding the step whose `inputField.name === "A"` (or "B") and reading its `inputValue`. They are NOT stored as a separate `p.measurements` field — the checklist is the single source of truth. **Back-compat (Rev A → installer-facing in Rev B).** Edit Pole modal in Rev A continues to render flat step list. To keep flat rendering working, Rev A's `renderPoleChecklist` is updated to flatten `sections[].steps[]` into a single ordered list when sections-format is detected, falls back to legacy `template.steps` when sections-format is absent. No installer behavior change in Rev A. **Migration strategy.** Existing Firebase data at `autoflag/procedures/{type}` may have legacy shape `{ steps: [...] }` or new shape `{ sections: [...] }`. The admin UI Rev A includes a one-click "Migrate to sectioned format" button per template that converts legacy `steps[]` → `sections: [{ id:"onsiteInstall", title:"Onsite Installation", kind:"steps", steps:[...same...] }]`. Phil clicks once per template after Rev A deploys. No automatic migration on read — explicit and reversible. --- ### Resolved during 2026-05-07 (AF2) session **Internal procedure scope.** Clone External 4-section structure as starting point. Same Pre-Install Measurements (A and B captured identically — internal halyards still need halyard-travel and flag-height). Halyard Setup is similar but without swivel-clip work since internal poles don't have external clips. Onsite Install similar with internal-pole specifics. ~70% step overlap with External. Refined in admin UI as field experience reveals differences. **Cylinder procedure scope.** Preserve existing seed data, restructure into new sectioned schema. Cylinder template has **only the `onsiteInstall` section** — Pre-Install Measurements and Halyard Setup sections are **omitted entirely** (not present as empty `{steps:[]}`). Renderer skips any section not present in the template. Admin UI can offer "Add section" for any type that later needs one. **Reference photos.** Reserve `step.referencePhotos: []` slot in the schema now. Admin UI in Rev A supports uploading them. Actual photos delivered by Phil later (Rev E). **Pre-install / status mapping.** No new pole statuses. Pole status enum stays as-is (`in_review → scheduled → in_progress → completed → verified`) and reflects overall job state. Section progress (which sections complete, which step in current section) is rendered from the checklist itself. No status change when measurements are captured. **Installs button styling.** Standard active-tab highlight, same treatment as Browse / Kanban / List / Map / Expand. Not always orange. Active = highlighted, inactive = muted. **A/B display.** Both. Substitution in Halyard Setup step text via `{A}`/`{B}` placeholders, AND a small reference pill at top of the install-steps section in the Edit Pole modal showing "A = 20 ft · B = 36 in" so installer doesn't have to scroll back to Pre-Install Measurements to see them. --- ## Build sequencing (proposed, not committed) Each rev is independently shippable. **Rev A — Data model + admin UI (no installer-facing changes yet)** — ✅ **SHIPPED Rev 1.89** - Extend `procedures` schema to support sections + reference content + per-step required-photo flag + per-step input field for A and B - Update `openProceduresModal()` admin UI to author sections + steps + per-step required photo + per-step input fields (limited to A and B) - Migrate `seed_procedures.json` to new schema, re-author External per the PDF Phil sent (full 4 sections), populate Internal as External clone for now, restructure Cylinder - No changes to Edit Pole modal yet — it keeps reading the flat step list during this rev (back-compat) **Rev B — Edit Pole modal renders sectioned procedure** — ✅ **SHIPPED Rev 1.90** - Replace flat step list in Edit Pole modal with sectioned UI - Terminology collapsible at top - Three step sections below as expandable groups (collapsed by default, tap to open) — auto-expand sections containing the active step - Per-step photo requirement enforced (block-until-photo-uploaded) - Per-step input fields (A/B measurements) captured with debounced save + visual confirmation - Forced sequencing across all steps in all sections — server-side guard in `toggleChecklistStep` plus UI lock visualization (.locked + .next-up styles) - Reference pill at top showing A and B once captured (or "not yet captured" placeholder) - {A} and {B} substitution in step instructions via `substituteMeasurements()` (XSS-safe, escapes-then-substitutes) - Continuous step numbering across sections (1..N globally) - Steps with `inputField` show 📝A / 📝B badge;
